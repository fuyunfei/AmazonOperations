## 核心问题：当前报表结构对数据流不友好

用户现在下载的是**聚合窗口快照**（7天/30天的汇总值），这对数据系统来说是一个根本性的设计矛盾：

```
4月9日下载"最近7天" → 包含 4月3日~4月9日的聚合数据
4月10日下载"最近7天" → 包含 4月4日~4月10日的聚合数据
                         ↑ 重叠6天，无法直接拼接成连续时间序列
```

ACOS、CTR、CVR 这类比率指标尤其不能直接相加，必须用原始计数反推。

---

## 正确的数据流设计

### 输入层：只摄入「单日快照」

```
每日输入：
  产品报表-子ASIN视图（昨天）  ← 唯一需要每日摄入的报表
  库存报表（有更新时摄入）

存储的是原始计数字段，不是比率：
  ✅ 广告花费、广告销售额、点击量、曝光量、订单量、GMV
  ❌ 不存 ACOS、CTR、CVR、ROAS（这些由系统计算）
```

### 存储层：按 ASIN × 日期 累积

```
time_series_table:
  asin | date | site | gmv | orders | ad_spend | ad_sales | clicks | impressions | ...
  
inventory_table:
  asin | snapshot_date | available_days | sellable_qty | ...
```

### 计算层：系统自行聚合任意时间窗口

```
7天ACOS  = SUM(ad_spend, 7天) / SUM(ad_sales, 7天)
30天TACoS = SUM(ad_spend, 30天) / SUM(gmv, 30天)
周环比    = 本周GMV / 上周GMV - 1
```

用户不需要再上传"最近7天"和"最近30天"两份报表，系统从每日积累的数据自动推算。

---

## 其他报表的数据流

单日快照无法覆盖的报表，采用**按需摄入**模式：

| 报表 | 摄入频率 | 存储方式 |
|---|---|---|
| 搜索词重构 | 每周上传一次 | 覆盖存储，保留最新版本 |
| 广告活动重构 | 每月上传一次 | 覆盖存储 |
| 库存报表 | 有变化时上传 | 带时间戳存快照 |
| 成本管理 | 价格/成本变动时更新 | 覆盖存储 |
| ABA搜索词对比 | 每月从Seller Central上传 | 带月份标签存档 |

---

## 数据流全景

```
[Nordhive]                    [Seller Central]
    │ 每日: 单日ASIN快照             │ 每月: ABA报表
    │ 按需: 搜索词/广告/库存          │
    ↓                               ↓
[摄入层]  解析xlsx → 提取原始计数字段 → 校验完整性
    ↓
[存储层]  时间序列表(ASIN×日期) + 各维度最新快照表
    ↓
[计算层]  按需聚合任意时间窗口 → 计算比率指标 → 阈值比对
    ↓
[输出层]  Alert触发 / 分析报告 / SOP建议
```

这样每天系统只需要摄入**1份报表**（单日ASIN快照），历史数据自动累积，任何时间窗口的分析都由系统计算，不依赖用户手动选择下载什么时间范围。
