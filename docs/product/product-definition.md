# AmazonAgent 产品定义

---

## 1. 产品目标

通过广告决策，缩小 SKU 运营目标与当前表现之间的差距。

```
用户的 SKU 运营计划（OP）:
  目标销量    = 300 件/月
  目标均价    = $45
  目标毛利率  = 20%

当前轨迹:
  当前日销    = 5 件（需要 10 件才能达标）
  当前 ACOS   = 35%
  当前毛利率  = 12%

Gap = OP 目标 - 当前轨迹

产品的作用: 诊断 Gap 的原因，输出广告层面能做什么来缩小它
```

广告是缩小 Gap 的杠杆之一。产品不替用户做所有决策，但在广告这根杠杆上，告诉用户：该加力、该收力、还是该换方向。

- 销量缺口 → 可能是流量不足（加预算/拓词）或转化率低（关键词不匹配）
- 利润缺口 → 可能是 ACOS 超盈亏线（砍无效词）或 CPC 过高（调出价）
- 增长健康 → 自然单占比在上升吗？BSR 在改善吗？

## 2. 核心洞察

**真正稀缺的不是数据，是判断。**

Amazon 广告后台有几千行数据。Helium 10、Jungle Scout 给你更多。数据从来不缺。

缺的是一个高级运营的判断力：看到同样的数据，知道什么该停、什么该加、什么再等等。这个判断力在一个人的脑子里 — 他走了，判断力就没了。


高级运营做判断时，脑子里有三样东西：

1. **业务目标**：这个产品要做到月销 300 件、毛利 20%，现在差多少
2. **经济参数**：盈亏平衡 ACOS 是多少，过了这条线就是亏
3. **阶段意识**：新品期 ACOS 200% 可以接受，成熟品不行；新关键词跑 3 天没曝光要查原因，老关键词 14 天没转化要停

Amazon 后台只给你当前状态。没有一个工具把这三样东西和数据放在一起做判断。现有工具对所有产品用同一套规则，因为它们不知道每个产品的目标、经济参数和阶段。

这导致两个可观察的现象：

**决策疲劳**是没有决策框架的症状。不知道目标和阶段，每一行数据都"可能重要"。有了框架，95% 的数据不需要看 — 没有穿越任何阈值，不需要行动。

**跨数据源分析**是做判断的前提条件，不是功能卖点。诊断"销量为什么没达标"需要交叉广告数据 + 销售数据 + 经济参数 + BSR 趋势。单一数据源无法诊断。

**AmazonAgent 的核心：把高级运营的判断逻辑编码成系统 — 基于用户的业务目标、产品经济参数和所处阶段，从数据中生成决策信号。人走了，判断力还在。**

## 3. 用户价值

产品的价值 = OP 目标达成率的提升 + 广告浪费的减少。

两个方向：

- **缩小 Gap**：诊断"为什么没达标"并给出广告层面的动作 — 流量不足则拓词加预算，转化率低则排查关键词匹配度，ACOS 超线则砍无效支出
- **减少浪费**：识别正在亏钱的广告支出（零成交词、ACOS 超盈亏线的广告组），量化每周损失金额

每条信号都有明确的美元金额。用户的价值感知不是"我用了一个工具"，而是"这条信号帮我发现日销差 5 件是因为广告流量不够"或"这条信号帮我每周省了 $200"。

更深层的价值：**不再依赖某一个高级运营的个人经验。** 判断逻辑在系统里，人员流动不影响决策质量。

## 4. 用户与场景

不按职位定义用户，按场景定义：

> **任何需要判断"这笔 Amazon 广告费该不该继续花"的人。**

具体场景：

| 场景 | 典型用户 | 使用方式 |
|------|---------|---------|
| 自己管广告的个体卖家 | 年销 $10-50 万，没有运营团队 | 每周看一次信号，直接操作 |
| 管多个产品的运营主管 | 带 3-10 人团队，负责广告预算 | 每周看信号，分配任务给运营执行 |
| 不懂广告但为广告付钱的老板 | 有运营团队，但不信任"一切都正常" | 每周 10 分钟看 P0 信号，拍板大决策 |
| 代运营服务商 | 管多个客户账号 | 用信号向客户汇报和建议 |

共同特征：**对广告花费有决策权或问责权，但没有时间逐行看数据。**

## 5. 信号输出格式

### 5.0.1 决策信号卡片

系统的核心输出单元。每个信号代表一个需要老板注意的事项。

```
┌─────────────────────────────────────────────┐
│  P0  零成交高消耗关键词                         │
│                                             │
│  描述: "portable air pump" 本周消耗 $47.20    │
│        但零成交，持续 3 周                      │
│                                             │
│  证据表: 关键词 | 消耗 | 点击 | 成交 | ACOS    │
│          ...    | $47  | 23  | 0   | ∞      │
│                                             │
│  财务影响: 每周损失 $47.20                      │
│  推荐操作: [暂停关键词] [降低竞价至 $0.30]       │
└─────────────────────────────────────────────┘
```

### 5.0.2 信号优先级

| 优先级 | 含义 | 期望处理时间 | 典型场景 |
|--------|------|------------|---------|
| **P0** | 紧急，正在漏钱 | 今天 | 零成交高消耗词、ACOS 远超盈亏平衡 |
| **P1** | 重要，需要本周处理 | 本周内 | 高 CVR 搜索词未提升匹配类型 |
| **P2** | 可优化 | 两周内 | 竞品做 Deal 分流、结构调整建议 |
| **P3** | 机会 | 有空时 | 可放大的高 ROI 关键词 |
| **good** | 积极信号 | 继续保持 | 自然订单占比上升、BSR 改善 |

### 5.0.3 健康评分 (Health Score)

对产品广告运营状态的综合评分，包含 5 个维度：

| 维度 | 评估内容 |
|------|---------|
| **conversion** | 转化率是否健康 |
| **budget_efficiency** | 预算利用效率，零成交消耗占比 |
| **traffic_quality** | 流量质量，CTR 和搜索词相关性 |
| **keyword_structure** | 关键词结构合理性（精确/短语/广泛配比） |
| **ad_efficiency** | 广告整体效率，ACOS vs 目标 |

## 5. 决策模型

### 5.1 三个输入

决策模型需要三个输入，缺一不可：

**输入 1：盈亏平衡线（系统计算）**

```
盈亏平衡 ACOS = (售价 - 成本 - FBA费 - 佣金) / 售价
```

客观的数学事实。过了这条线，每多花一块钱广告费就是净亏损。

**输入 2：ACOS / ROAS 目标（用户设定）**

用户的广告效率目标，反映其利润率期望。通常低于盈亏线。

**输入 3：SKU 运营计划 — OP（用户设定）**

```
OP = {目标月销量, 目标均价, 目标毛利率}
```

这是广告决策的业务锚点。没有 OP，系统只能告诉你"ACOS 太高"；有了 OP，系统能告诉你"**日销差 5 件是因为广告流量不足，当前 50 次点击/天 × 10% CVR = 5 单，目标 10 单需要 100 次点击**"。

三个输入的关系：

```
OP（业务目标）→ 决定广告需要完成什么
盈亏线（经济约束）→ 决定广告不能超过什么
ACOS 目标（效率期望）→ 决定广告应该做到什么
```

### 5.2 分析粒度

信号在**广告组级别**生成，下钻到**关键词级别**提供证据。

为什么不在关键词级别直接判断：一个关键词可能亏钱，但它为同广告组内的其他关键词引流（广泛匹配发现搜索词 → 精确匹配收割）。孤立看关键词会误杀有价值的引流词。

为什么不在活动级别：粒度太粗，掩盖了内部差异。

广告组是广告结构中"策略意图一致"的最小单元 — 同一个广告组内的关键词共享匹配类型和出价策略，适合作为决策单元。

### 5.3 信号生成：从 Gap 诊断到广告动作

信号不是从 ACOS 阈值生成的，而是从**OP 目标与当前轨迹的 Gap**诊断出来的。

```
第一步：计算 Gap
  Gap(销量)  = 目标月销 - 当前轨迹月销
  Gap(利润)  = 目标毛利率 - 当前毛利率
  Gap(效率)  = 目标 ACOS - 当前 ACOS

第二步：诊断原因（广告能影响的）
  销量 Gap → 流量不足？（曝光/点击不够）→ 加预算、拓词
           → 转化率低？（CVR 低于品类均值）→ 关键词不匹配、Listing 问题（标记但不修）
  利润 Gap → ACOS 超盈亏线？→ 识别亏损广告组/关键词
           → CPC 过高？→ 出价策略调整
  效率 Gap → 零成交高消耗词？→ 否词/暂停
           → 高 ROI 词未扩量？→ 加预算/提升匹配类型

第三步：排优先级
  P0 = 正在亏钱（ACOS > 盈亏线 + 阶段未豁免），或阶段约束被违反
       → 不行动则持续损失，需要今天处理
  P1 = 有明确 Gap 且有可执行的广告动作
       → 本周处理
  P2 = 优化机会（可以更好，但不紧急）
       → 两周内
  good = 表现好于目标，值得扩大
       → 确认并寻找扩量空间

每条信号附带:
  - 证据：哪些关键词/广告组贡献了这个 Gap
  - 影响：每周美元金额（亏损或机会）
  - 趋势：在恶化还是改善
  - 动作：具体操作建议
```

### 5.4 两层阶段模型

阶段感知分两层：**产品阶段**决定整体策略，**关键词生命周期**决定单个关键词的评判标准。

#### 产品阶段（用户手动标记）

| 阶段 | Gap 诊断重心 | 约束 | 约束度量 | P0 触发 |
|------|-----------|------|---------|--------|
| **new** | 曝光和初单 | 总支出 < 预算上限 | 周支出金额 | 曝光 < 500/周；花钱了零展示 |
| **early** | 关键词发现和排名 | 单词浪费 < $20 | 零成交词累计消耗 | 零成交 > $20；高 CVR 词未升级 |
| **growth** | 缩小 OP Gap | BSR 不连续下降 | BSR 周环比 ≥ -5%（连续 2 周） | ACOS > 盈亏线；零成交 > $15 |
| **mature** | 利润最大化 | 自然单占比不降 + BSR 稳定 | 自然订单占比 ≥ 上月；BSR 周环比 ≥ -3% | ACOS > 目标；自然占比连降 2 周 |

阶段由用户手动标记 — 因为阶段是业务判断，不是数据事实。

#### 关键词生命周期（系统自动计算）

一个成熟产品上的新关键词，不能用成熟期的标准去评判。关键词自身也有生命周期：

| 阶段 | 时间 | 评判标准 | 信号 |
|------|------|---------|------|
| **发现期** | Day 1-3 | 有没有曝光？ | 零曝光 → 出价太低或词不匹配 |
| **验证期** | Day 4-7 | 有没有点击？CTR 如何？ | 曝光有但 CTR 极低 → 词与 Listing 不匹配 |
| **确认期** | Day 8-14 | 有没有转化？ | 有点击零转化 → 流量质量问题或价格/Listing 问题 |
| **优化期** | Day 15+ | 赚不赚钱？ | 适用标准的盈亏平衡评估 |

关键词生命周期由系统根据关键词激活日期自动计算，不需要用户标记。

**两层组合的效果**：一个 growth 阶段的产品上刚加了一个新关键词 — 产品用 growth 的约束（ACOS > 盈亏线 = P0），但这个关键词只跑了 3 天，还在发现期 — 此时不判断 ACOS，只看有没有曝光。两层逻辑避免了误杀正在测试中的新词。

## 6. 边界

### 做什么

- 接收用户的 SKU 运营计划（OP：目标销量、均价、毛利率）
- 读取 Amazon 广告数据（关键词、广告组、搜索词、销售）
- 结合产品经济参数计算盈亏平衡线
- 诊断 OP 目标与当前表现的 Gap，识别广告层面的原因和动作
- 基于产品阶段 + 关键词生命周期生成带优先级的决策信号
- 量化每条信号的财务影响（$）
- 提供决策闭环（确认 / 拒绝 / 推迟 + 备注）
- 追踪决策效果（确认后对比下周数据变化）

### 不做什么

| 不做 | 原因 |
|------|------|
| 自动执行（自动调价、自动暂停关键词） | 我们输出判断，执行由人完成。自动执行是另一个产品 |
| 库存管理 | 需要供应链数据（工厂交期、MOQ、物流），不在我们的数据范围内 |
| 自动改价 / Repricing | 实时竞价系统，基础设施问题，与决策信号是不同的产品形态 |
| Listing 优化 / 文案生成 | Amazon 自己在做，ChatGPT 能做，零壁垒 |
| 账号合规 / 申诉 | 法律专业领域，不是数据分析能解决的 |
| 多平台（Walmart / TikTok Shop） | 每加一个平台 = 全新的数据管道，先在 Amazon 做透 |
| 站外获客（Google Ads / 社交引流） | 目前没有数据源。SP-API 接通 Amazon Attribution 后可作为自然延伸 |
| 定义阶段 | 阶段是用户的业务判断，系统不替用户决定 |

### 一句话边界

**输入是广告数据，输出是决策信号，执行靠人。**
