> 参与者：傅云飞（我方）、Louie Liu（客户方）

## 核心发现

### 1. 客户的运营 SOP 流程

Louie 描述了完整的亚马逊运营 SOP：

1. **创建 Listing** → 与竞品对比多维度评估（能否达到竞品水平）
2. **积累 Review** → Review 积累过程需可控
3. **开广告** → 基于广告数据分析，有两条线：
   - 调整广告（否词、降预算等）
   - 广告数据分析发现问题（如 Claude 分析总是在让他否词）

### 2. 广告分析的具体痛点

- 现有 AI（Claude 直接使用）分析广告数据时，**只从词语维度或广告组维度做分析**
- **无法判断整体关键词池是否有问题**，也不会建议拓展新词
- 缺少**词库管理**的概念：哪些词已经用了、表现如何、哪些词没用、什么时候该用

### 3. 关键词的阶段性目标

Louie 提出关键词应该有**阶段性目标**：

> "比如说三天他应该达到一个流量目标，然后七天应该达到一个点击目标，然后十四天应该会有一个转化目标"

这与我们产品定义中的"阶段约束"概念高度吻合，但粒度更细 — 不只是产品级别的阶段，**关键词本身也有生命周期**。

### 4. Memory / Context 管理

- 客户需要系统记住**所有 AI 之前的决策**（词库、已跑过的数据、SOP 等）
- 这些信息放在 SQL 表格中，AI 在做判断时可以查询
- SOP 内容最终会变成可复用的 skill / scale 文档

### 5. Listing 分析

- Listing 拆解为：标题（有生成逻辑）、卖点（基于产品参数）、图片
- 图片分析：把已做好的图片喂给 AI，让 AI 判断"这张图要表达什么"，再与设计意图对比
- 用户对 AI 生成 Listing 内容**不满意** — "反正我己给到我的内容的话，我是没有那么满意的"

### 6. 交互形态偏好

- **不需要实时通讯**，每天推送通知即可
- **飞书 Bot** 推送，运营自行处理，"有点像 Todo list"
- 特殊情况（如 listing 被下架、有人跟卖）需要**实时告警**
- 日常以**日为单位**循环更新数据即可

### 7. 数据来源

- 目前**人工在收集数据**，所以是 Excel 表
- 两个数据来源：亚马逊后台 ABA 报告 + 第三方工具（西柚/卖家精灵反查数据）
- 已有一套**分词和打标逻辑**（词库管理的基础）

### 8. 关于本地模型 vs 云端

- 傅云飞认为现阶段**不需要本地部署模型**，纯骗钱
- Louie 认为 memory 和 scale（经验/流程）应该在**云端**，算力用通用模型即可
- 共识：**先产出价值，别纠结数据安全和模型训练**

## 对产品定义的验证与挑战

| 产品定义中的论断 | 验证/挑战 |
|-------------|---------|
| 用户需要决策信号而非更多数据 | **验证** — Louie 明确说 AI 分析"只会否词"，缺乏整体判断 |
| 阶段感知是核心差异化 | **验证并深化** — 不只是产品阶段，关键词本身也有阶段性目标 |
| 信号在广告组级别生成 | **挑战** — 客户需要"词库"级别的管理视角，跨广告组看关键词全貌 |
| 每天推送 vs 每周分析 | **修正** — 客户期望日频推送（飞书 Bot），不是每周看一次 |
| 不做 Listing 优化 | **确认** — 用户自己对 AI Listing 不满意，这不是我们该做的 |
